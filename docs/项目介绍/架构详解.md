# BuddyKnow — 个人碎片知识落库系统

> **一句话介绍**：把你随手看到的公众号文章、小红书笔记、知乎回答、PDF、图片、语音消息……自动变成一个「可搜索、可问答、自动归类」的私人知识库。

---

## 一、这个项目解决什么问题？

我们每天都在「碎片化消费」知识 —— 微信群里转一篇深度文章、小红书刷到一个实用方法论、知乎看到一个精彩回答。但几天后想找的时候，翻遍聊天记录也找不到了。

**BuddyKnow 做的事情**：把这些碎片知识「一键落库」，然后自动整理成 Wiki，你随时可以用自然语言提问，系统从你的知识库里找到答案。

**核心价值**：
- **存得进来**：支持微信公众号、小红书、知乎、任意网页、PDF、图片、语音等 7 种内容来源
- **找得出来**：不是简单的关键词搜索，而是「语义搜索 + AI 回答」，像问一个读过你所有收藏的助手
- **自动归类**：系统自动把你的知识编译成 Wiki 页面，按主题/人物/概念分类，形成你的个人知识图谱

## 三、数据流：一篇文章从「收藏」到「可问答」经历了什么？

```
用户发送链接
    │
    ▼
[1] 去重检查 ─── 已存在？→ 返回「已入库」
    │
    ▼
[2] 网页抓取 ─── 根据域名自动选择抓取器（微信/小红书/通用）
    │                │
    │                ├─ 微信公众号：BeautifulSoup 解析 HTML + 图片占位符
    │                ├─ 小红书：MCP 服务获取完整数据（降级 HTTP 抓取）
    │                └─ 通用网页：Readability 算法提取正文
    │
    ▼
[3] 图片 OCR ─── 有图片？→ Vision LLM 智能识别（分类→专用 Prompt）
    │               支持文字图、数据图表、表格、代码截图、照片
    │               图文穿插：图片内容插回原文对应位置
    │
    ▼
[4] LLM 清洗 ─── 只提取元数据（标题/摘要/标签），正文保留原文
    │               规则引擎做基础降噪（去广告/引导语/联系方式）
    │
    ▼
[5] Markdown 落库 ─ 保存为 data/YYMMDD_UUID_标题.md
    │                 YAML Front-matter 记录元数据
    │
    ▼
[6] 向量索引 ──── 文章切片 → Embedding 模型 → ChromaDB 存储
    │               同时构建 BM25 关键词索引
    │
    ▼
[7] Wiki 编译 ─── 新文章 + 现有 Wiki 摘要 → LLM 决定创建/更新哪些页面
                    按主题(topic)/人物(entity)/概念(concept) 分类
                    自动生成 MOC 导航页 + 交叉引用
```

## 四、模块详解

### 4.1 配置层 — `config.py`

**做什么**：从 `.env` 文件读取所有配置项（API Key、模型选择、端口等），给全局使用。

**关键设计**：
- LLM Provider 可插拔：支持 Kimi（月之暗面）、DeepSeek、Ollama（本地模型），切换只需改一行 `.env`
- 所有 Provider 统一走 OpenAI 兼容接口，代码不需要改

**为什么这么设计**：不同场景用不同模型。日常入库用便宜的 Kimi/DeepSeek，本地测试可以用免费的 Ollama。

---

### 4.2 入库管线 — `services/ingest_pipeline.py`

**做什么**：统一入库入口。不管你是从网页、微信、CLI 哪个入口进来的，都走这个管线的 7 步流程。

**关键设计**：
- **去重检查**：相同 URL 不会重复入库
- **强制重入库**：`force=True` 时删旧文件重来，保证内容最新
- **进度回调**：支持 SSE 流式推送入库进度（抓取中 → OCR 中 → 清洗中 → 完成）
- **Wiki 编译入队**：入库完不阻塞，把编译任务丢进队列后台处理

**为什么这么设计**：之前各个入口（CLI、企微、Web）各自写入库逻辑，导致代码重复且行为不一致。统一管线后，新增入口只需要调一行 `ingest_url()`。

---

### 4.3 内容抓取层 — `ingestion/`

这个模块负责「从各种来源把内容抓回来」，是整个系统的「嘴巴」。

#### 4.3.1 抓取器基类 — `base.py`

定义了统一接口 `BaseFetcher`，所有抓取器都继承它，输出统一格式 `RawContent`（标题、正文、作者、来源平台、图片列表、原始标签）。

就像不同餐厅的菜最后都装进同一个标准餐盒。

#### 4.3.2 抓取器路由 — `router.py` + `dispatcher.py`

**做什么**：根据输入的 URL 或文件类型，自动选择对应的抓取器。

```
URL → 微信域名？→ WechatFetcher
    → 小红书域名？→ XiaohongshuFetcher
    → 其他？→ GeneralFetcher
文件 → .pdf？→ PDFParser
     → .jpg/.png？→ VisionOCR
     → .mp3/.m4a？→ AudioTranscriber
```

#### 4.3.3 微信公众号抓取器 — `wechat.py`

**做什么**：专门解析微信公众号文章。

**特殊能力 — 图文穿插**：
- 微信文章经常图文交替出现，普通抓取器会把文字和图片分家
- 这个抓取器用 `[IMG_1]` `[IMG_2]` 占位符标记图片位置
- 后续 OCR 完成后，图片内容会插回原文对应位置
- 效果：一篇「5 个数据图表解读行业趋势」的文章，图表数据不会丢失

**技术细节**：微信文章是服务端渲染（不需要 JS 执行），所以用 requests + BeautifulSoup 就够了，不用上 Headless Browser。

#### 4.3.4 小红书抓取器 — `xiaohongshu.py`

**做什么**：抓取小红书笔记内容。

**双模式设计**：
- **MCP 模式**（优先）：通过 Docker 部署的 xiaohongshu-mcp 服务获取完整数据（标题、正文、图片、标签、视频封面），稳定可靠
- **HTTP 降级**：MCP 不可用时，直接请求小红书页面，从 SSR 数据（`window.__INITIAL_STATE__`）中提取内容，但容易受反爬限制

**视频帖处理**：小红书很多内容是视频。系统会提取视频封面图做 OCR，把视频时长信息也保留下来。

#### 4.3.5 通用网页抓取器 — `general.py`

**做什么**：处理知乎、博客、新闻等常规网页。

**技术方案**：Readability 算法（Mozilla 开源）提取页面主要内容，自动去掉导航栏、侧边栏、广告等噪声。

**知乎特殊处理**：
- 优先走知乎 API 获取回答/文章内容
- API 被限流时，从页面 SSR 注入的 `js-initialData` JSON 中提取
- 最后兜底用 Readability

#### 4.3.6 PDF 解析器 — `pdf_parser.py`

**做什么**：解析 PDF 文件，提取文字内容。

**双策略**：
- **文本型 PDF**（大部分电子书、文档）：pymupdf 直接提取，速度快
- **扫描型 PDF**（图片转的 PDF）：检测到文本过短时自动降级到 Vision OCR —— 每页渲染成图片 → LLM 识别文字

#### 4.3.7 图片 OCR — `vision_ocr.py`

**做什么**：用 Vision 大模型（Kimi Vision）识别图片中的内容。

**不是简单的 OCR**，而是「智能图片理解」：
1. **先分类**：判断图片是文字图、数据图表、表格、代码截图、还是照片
2. **再用专用 Prompt**：
   - 文字图：提取所有文字，保留排版结构
   - 数据图表：分析趋势，提取数据标签和数值，输出 Markdown 表格
   - 表格：还原为 Markdown 表格格式
   - 代码截图：还原完整代码
   - 照片：描述核心内容
3. **多图关联**（≤3张）：多张图一起发给 Vision 模型，理解图间叙事关系

**成本**：约 0.01 元/张，零内存开销（调用 API 而非本地模型）。

#### 4.3.8 音频转录 — `audio_transcriber.py`

**做什么**：把语音消息转成文字。

- 使用 OpenAI Whisper API
- 大文件（>25MB）自动按 10 分钟分段处理
- 成本：约 2.5 元/小时音频

---

### 4.4 LLM 清洗 — `transform/llm_cleaner.py`

**做什么**：把抓回来的原始内容「规范化」。

**重要设计决策**：LLM **只负责提取元数据**（标题、摘要、标签），**不重写正文**。

为什么？
- 重写正文会丢失信息（案例、数据、论述细节）
- Token 成本高（长文要多次调用）
- 原文直接落库，检索时能命中完整上下文

**规则引擎降噪**（不用 LLM，零成本）：
- 去掉公众号引导语（"点击关注""记得点赞"）
- 去掉联系方式（微信号、公众号名）
- 去掉版权声明、招聘广告
- 去掉重复段落（微信抓取常见问题）

---

### 4.5 存储引擎 — `storage/markdown_engine.py`

**做什么**：把清洗后的知识存为本地 Markdown 文件。

**为什么用 Markdown 而不是数据库**：
- 可读性：直接用文本编辑器打开就能看
- 可移植：不依赖任何数据库系统
- Git 友好：可以版本管理
- AI 友好：LLM 原生理解 Markdown

**文件命名规范**：`YYMMDD_短UUID_精简标题.md`

**文件格式**：
```markdown
---
title: "文章标题"
summary: "一句话摘要"
tags: ["标签1", "标签2"]
source_url: "https://..."
source_platform: "wechat"
author: "作者名"
created_at: "2026-04-19 12:00:00"
---

# 文章标题

> **摘要**: 一句话摘要

（原始正文，完整保留）
```

### 4.6 检索系统 — `retrieval/`

这是系统的「大脑」，负责理解你的问题并从知识库中找到答案。

#### 4.6.1 语义切片器 — `chunker.py`

**做什么**：把长文章切成小块（chunks），方便检索。

**切片策略**：
- 按 `##` / `###` 标题层级切（保证语义完整）
- 没有标题的文章按约 2000 字切，相邻切片有 300 字重叠（防止语义断裂）
- 每篇文章额外生成一个「摘要切片」，包含标题+摘要+标签，提高泛化检索命中率

#### 4.6.2 向量索引器 — `indexer.py`

**做什么**：把每个文本切片变成向量（一串数字），存入 ChromaDB。

**为什么需要向量**：向量能表示语义。两段意思相近但用词不同的文字，向量距离很近。比如搜「AI 怎么改变产品经理工作」，能命中一篇标题叫「大模型时代的产品思维转型」的文章。

**模型选择**：`BAAI/bge-small-zh-v1.5`（中文优化，轻量级）
- bge 系列需要在查询时加 instruction 前缀来区分 query/document 模式
- 系统自动处理了这个细节

#### 4.6.3 BM25 关键词索引 — `bm25.py`

**做什么**：传统的关键词搜索（类似搜索引擎的匹配原理）。

**为什么需要它**：向量搜索擅长理解语义，但有时候你就是想搜一个具体的人名、公司名、术语，这时候关键词搜索更准。

**特殊优化**：标题 3 倍加权 —— 如果标题里出现了搜索词，说明这篇文章高度相关。

**分词**：jieba 精确模式 + 自定义词典 + 停用词过滤。

#### 4.6.4 Cross-Encoder Reranker — `reranker.py`

**做什么**：对初步检索结果做「二次精排」。

**为什么需要它**：向量搜索和 BM25 各自返回了一批「可能相关」的结果，但排序不够精准。Reranker 拿到 (问题, 文本片段) 对，用 Cross-Encoder 模型重新打分，把最相关的排到前面。

**模型**：`BAAI/bge-reranker-v2-m3`（多语言，约 570MB）
- 延迟加载：首次搜索时才加载模型，不浪费启动时间
- 常驻内存：加载一次后不释放，后续搜索直接用

#### 4.6.5 混合搜索引擎 — `hybrid_searcher.py` ⭐核心

**做什么**：把上面的所有检索能力组合起来，完成「问题 → 答案」的全流程。

**搜索流程**：

```
用户提问
    │
    ▼
[1] QA 缓存回查 ── 之前有人问了相同问题并且标记了「有帮助」？→ 直接返回缓存答案
    │
    ▼
[2] 意图分类 ──── 规则判断（零 LLM 调用）：
    │               细节查找 / 精确查找 / 主题概览 / 实体查询 / 模糊查询 / 否定查询
    │
    ▼
[3] 并行执行（两个任务同时跑，节省时间）：
    │               ├─ [3a] 原始 query 在向量库 + BM25 中检索
    │               └─ [3b] LLM 改写 query 为 2-3 个精确检索词
    │
    ▼
[4] 补充检索 ──── 用改写后的 query 再检索一轮，合并结果
    │
    ▼
[5] RRF 融合 ──── 把向量搜索和 BM25 的结果合并排序
    │               摘要切片 1.3x 加成，双路命中 1.2x 加成
    │               根据意图类型调整各路权重
    │
    ▼
[6] Reranker 精排 ─ Top 20 候选 → Cross-Encoder 精排 → Top 5
    │
    ▼
[7] LLM 生成回答 ─ 把检索到的内容 + 用户问题发给 LLM
    │                 注入 Few-shot 示例（用户认可的好回答）
    │                 动态 max_tokens（内容多就答长点）
    │
    ▼
[8] 后处理 ───── 提取高亮关键词 / 缓存优质回答 / 异步回填候选
```

**关键设计**：

- **意图路由 + 分路加权**：不同意图下，对 Wiki 页面和原始文章的重视程度不同
  - 查具体细节：原始文章权重 2.0（要精确事实）
  - 查主题概览：概念卡权重 2.0（要宏观理解）
  - 查人物/公司：实体页权重 3.0（要结构化事实）

- **Few-shot 注入**：系统会读取用户之前标记「有帮助」的问答对，作为示例注入 Prompt，让 LLM 参考这些「好回答」的风格和深度

- **QA 缓存复利**：高质量回答自动缓存到文件，下次有人问类似问题直接返回，不浪费 LLM 调用

- **流式输出**：`search_stream()` 支持 SSE（Server-Sent Events），前端可以逐字显示答案，不用等全部生成完

### 4.7 Wiki 编译系统 — `wiki/`

这是系统的「知识整理员」，负责把碎片知识编织成结构化的 Wiki。

#### 4.7.1 编译引擎 — `compiler.py`

**做什么**：每次新文章入库后，分析新文章 + 现有 Wiki 状态，决定需要创建/更新哪些 Wiki 页面。

**编译流程**：
1. 读取新文章内容
2. 获取现有 Wiki 所有页面的轻量摘要
3. 用 Embedding 找到相似度 ≥ 85% 的已有页面（语义去重）
4. 把以上信息发给 LLM，让它输出「编译计划」（JSON 格式）：
   - `new_pages`: 需要创建的新页面
   - `update_pages`: 需要更新的已有页面（追加/标记矛盾）
   - `cross_references`: 页面间的交叉引用
5. 执行编译计划：创建新页面或向已有页面追加洞察

**页面类型**：
- **topic**（主题页）：围绕一个概念/方法论/技术/趋势
- **entity**（实体页）：围绕一个具体的人物/公司/产品
- **concept**（概念卡）：一个核心概念的精炼定义

**关键约束**：一篇文章最多创建 1-2 个新页面，防止过度拆分。

#### 4.7.2 编译队列 — `compile_queue.py`

**为什么需要队列**：
- 入库可以多线程并发（你同时发 5 个链接进来）
- 但 Wiki 编译必须串行（不然两个 LLM 同时改同一个页面会冲突）
- 解决方案：asyncio.Queue + 单消费者协程

#### 4.7.3 分类树 — `taxonomy.py`

**做什么**：维护 `wiki/_taxonomy.yaml`，一个自动演化的分类体系。

**自演化机制**：
- 每次编译新页面后，LLM 决定这个页面属于哪个分类
- 分类不存在就新建，分类下页面太少就合并
- 上限 12 个一级分类，超过触发自动合并
- 分类体系随着你的知识库内容动态生长

#### 4.7.4 MOC 导航页 — `moc_builder.py`

**做什么**：为每个分类生成一个「Map of Content」导航页，串联该分类下的所有页面。

比如分类「AI 应用」下面的 MOC 页面会列出：Vibe Coding、AI 产品设计、Prompt Engineering 等所有相关页面的链接和摘要。

**零 LLM 调用**：纯代码遍历生成，100% 准确。

#### 4.7.5 页面存储 — `page_store.py`

Wiki 页面的 CRUD 操作。支持 Append-only（只追加不删除）模式 —— 新知识追加到页面的「新增洞察」区域。

#### 4.7.6 健康检查 — `inspector.py`

定期检查 Wiki 健康状态：
- 孤立页面（没有任何页面引用它）
- 缺失引用（被引用但实际不存在的页面）
- 过时页面（超过 30 天没更新）
- 内容质量（空页面、超短页面）

#### 4.7.7 索引生成 — `index_builder.py`

自动生成 `wiki/_index.md`，列出所有 Wiki 页面的目录。零 LLM 调用，纯遍历生成。

---

### 4.8 AI 助手 — `assistant/`

**做什么**：提供一个可以「聊天」的智能助手，支持多轮对话。

#### 4.8.1 意图识别 — `intent.py`

分析用户发来的消息，判断他想干什么：
- 含 URL → 入库
- 含「搜索/查询/有没有」→ 搜索知识库
- 含「统计/状态/多少篇」→ 系统查询
- 其他 → 闲聊

**纯规则匹配**，不调用 LLM，零延迟。

#### 4.8.2 对话引擎 — `chat_engine.py`

基于 OpenAI 接口的多轮对话：
- 会话存储在内存中（1 小时过期，最多 20 轮）
- 支持流式输出（SSE）
- 支持上下文注入（搜索结果、系统状态等）

#### 4.8.3 API 路由 — `router.py`

提供 HTTP 接口：
- `POST /api/assistant/chat` — 流式对话
- `POST /api/assistant/chat-sync` — 非流式对话
- `GET /api/assistant/sessions` — 会话列表
- `DELETE /api/assistant/sessions/{id}` — 清空会话

#### 4.8.4 定时任务 — `scheduler.py`

后台定时任务：
- 每周知识摘要推送
- 知识回顾推荐
- 编译状态通知
- 支持用户订阅/取消订阅推送

### 4.9 微信集成

#### 4.9.1 微信 iLink Bot — `ilink/`

**做什么**：通过微信个人号机器人接收消息，直接在微信里「转发即入库」。

**工作方式**：
- 基于 `wechat-ilink-bot` SDK
- 你在微信里转发一篇文章给 Bot → 自动入库
- 你在微信里发文字问题 → 自动搜索回答
- 支持 Token 持久化 + 断线自动重连

#### 4.9.2 企业微信回调 — `wecom/`

**做什么**：通过企业微信自建应用实现消息收发。

**模块**：
- `callback.py` — 消息接收 + 后台处理（入库/搜索/对话）
- `crypto.py` — 企微消息加解密
- `sender.py` — 主动推送消息给用户

**消息类型支持**：
- 文本消息：识别 URL → 入库，普通文字 → 搜索/对话
- 链接消息：直接入库
- 图片消息：Vision OCR → 入库
- 语音消息：Whisper 转录 → 入库
- 文件消息：按类型分发（PDF/音频等）
- 视频消息：暂不支持（提示用户发链接）

**防重机制**：消息去重缓存（MsgId → 时间戳），5 分钟 TTL，防止企微重试导致重复处理。

## 五、技术选型与演变

### 5.1 为什么选这些技术？

| 组件 | 选型 | 原因 |
|------|------|------|
| Web 框架 | FastAPI | 异步支持好、自动生成 API 文档、性能高 |
| 向量数据库 | ChromaDB | 纯 Python、嵌入式（无需独立服务）、轻量 |
| Embedding 模型 | bge-small-zh-v1.5 | 中文优化、轻量（~100MB）、效果够用 |
| Reranker 模型 | bge-reranker-v2-m3 | 多语言支持、精排效果好 |
| 关键词搜索 | jieba + 手写 BM25 | 轻量、可控、支持自定义词典 |
| LLM | DeepSeek / Kimi | 便宜（DeepSeek-Chat 约 ¥1/百万 token）、中文能力强 |
| OCR | Kimi Vision API | 零部署、支持图表理解、便宜 |
| 语音转录 | OpenAI Whisper API | 效果最好、便宜（¥2.5/小时） |
| PDF 解析 | pymupdf | 纯 CPU、轻量、有预编译 wheel |
| 前端 | Alpine.js + Tailwind | 单文件 SPA、零构建、够用 |
| 部署 | Docker Compose | 一键启动（主服务 + 小红书 MCP 服务） |

### 5.2 关键设计决策的演变

| 决策 | 早期方案 | 现在方案 | 为什么改 |
|------|---------|---------|---------|
| 正文处理 | LLM 重写脱水 | 只提取元数据，保留原文 | 知识零丢失，Token 成本降 90% |
| 搜索架构 | 纯向量搜索 | 向量 + BM25 + Reranker 三路 | 关键词搜索补充精确匹配场景 |
| 入库流程 | 各入口各自实现 | 统一管线 `ingest_pipeline` | 消除代码重复，行为一致 |
| 单例管理 | 到处 new 对象 | `main.get_*()` 全局单例 | 避免重复加载模型（Embedding ~100MB、Reranker ~570MB） |
| Wiki 编译 | 同步执行 | 异步队列串行 | 避免并发写冲突 |
| 反馈机制 | 点赞/踩无实际作用 | 反馈 → QA 缓存 + Few-shot 注入 | 形成闭环，越用越准 |
| 微信接入 | 企业微信自建应用 | 微信 iLink Bot（个人号） | 更轻量，不需要企业微信管理员权限 |
| 生命周期管理 | `@app.on_event("startup")` | `lifespan` 上下文管理器 | 前者已被 FastAPI 标记为废弃 |

## 七、反馈闭环：系统如何「越用越聪明」

```
用户提问 → 系统回答 → 用户点「有帮助」👍
    │                           │
    │                           ▼
    │                    1. 记录到 feedback.jsonl
    │                    2. 自动写入 QA 缓存（下次相同问题直接返回）
    │                    3. 下次搜索时注入为 Few-shot 示例
    │                           │
    ▼                           ▼
  LLM 看到好回答示例 → 生成更高质量的回答 → 更多用户满意 → ...
```

这个闭环让系统不是静态的，而是随着使用不断积累优质回答模式。
