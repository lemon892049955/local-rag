# BuddyKnow 项目介绍（产品经理视角）

---

## 一、简历版（3-4 行，直接贴简历）

**BuddyKnow — 个人碎片知识管理与 RAG 问答系统** | 独立负责 | 2026.04

从 0 到 1 设计并落地一套 AI 驱动的个人知识库产品，覆盖"采集→清洗→编译→检索→问答"全链路。支持 6 大数据源（公众号/小红书/知乎/PDF/图片/音频）一键入库，LLM 自动编译为结构化 Wiki 知识网络；基于混合检索（BM25 + 向量 + RRF 融合 + Cross-Encoder 精排）实现 RAG 问答，Recall@5 从 41% 优化至 82%，Hit Rate 从 50% 提升至 91%。企微"转发即入库"实现零门槛使用，月度运营成本 ¥17。独立完成需求定义、架构设计、全栈开发、效果评测及 4 轮迭代优化。

## 二、完整版（面试阐述用）

### 2.1 需求洞察

**痛点发现**

作为产品经理，我每天刷公众号、小红书、知乎等平台会看到大量有价值的行业文章，但存在三个核心问题：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **碎片化严重** | 好文章散落在 5+ 个平台，收藏后再也不看 | 知识积累为零 |
| **检索困难** | "之前看过一篇讲 Vibe Coding 的文章"→ 翻遍收藏也找不到 | 需要时用不上 |
| **缺乏结构** | 看了 30 篇产品经理面试文章，脑子里还是一团浆糊 | 无法形成体系 |

**解决思路**

不做又一个"收藏夹"，而是做一个**会自动整理的第二大脑**：
- 入库零门槛：微信转发就行，不用切 App
- AI 自动整理：LLM 清洗降噪 + 编译成结构化 Wiki
- 随时问答：自然语言提问，秒级返回带引用的答案

**目标用户**：信息密度高的知识工作者（产品经理/研究员/投资人）

### 2.2 架构设计

**系统总览**

```
用户触点                    核心引擎                     数据层
───────                    ────────                    ──────
微信转发 ─┐                                           ┌─ data/ (原始文章)
Web 界面 ─┤   采集 → 清洗 → 落库 → 索引 → 编译       ├─ wiki/ (知识网络)
CLI 命令 ─┘   ↑                         ↓             └─ vectordb/ (向量索引)
              6大数据源              Wiki编译引擎
              解析器                 (LLM持续编译)
                    ↘              ↙
                  混合检索 + RAG 问答
                  (BM25+向量+RRF+Reranker)
```

**关键设计决策**

| 决策点 | 选择 | 为什么 |
|--------|------|--------|
| 数据格式 | Markdown + YAML | 兼容 Obsidian/Logseq，可迁移，不锁定 |
| 双层架构 | data/(不可变) + wiki/(追加) | 原始来源不丢失，Wiki 可重建 |
| 检索策略 | 混合检索而非纯向量 | 纯向量对中文关键词匹配差，BM25 补位 |
| LLM 策略 | 兼容 3 家（Kimi/DeepSeek/Ollama） | 避免供应商锁定，成本可控 |
| 企微接入 | 自建应用而非机器人 | 支持回调、多消息类型、主动推送 |
| Wiki 编译 | Append-only + 语义去重 | 非破坏性更新，同主题自动合并 |

**核心模块**

| 模块 | 功能 | 规模 |
|------|------|------|
| 多源采集层 | 公众号/小红书(MCP)/知乎/PDF/图片OCR/音频转录 | 6 类解析器 |
| AI 清洗层 | LLM 降噪 + 长文分段清洗 + 图文穿插 OCR | 万字长文零丢失 |
| Wiki 编译引擎 | LLM 持续编译知识网络（主题页+实体页+交叉引用） | 29 篇→27 页+9 分类 |
| 混合检索层 | BM25 + 向量 + RRF 融合 + Cross-Encoder 精排 | 143 个切片，5 路召回 |
| RAG 问答 | 查询改写 + Wiki/Data 双路召回 + LLM 答案生成 | 24 个 API 端点 |
| 企微集成 | 转发即入库 + AI 对话 + 主动推送 | 支持文本/链接/图片/语音/文件 |
| Web 前端 | 知识图谱(D3) + 分类浏览 + AI 助手悬浮窗 | 零构建 SPA |
| 评测体系 | 四维度评测（忠实度/上下文/答案/延迟分段） | 60 条用例，LLM-as-Judge |

### 2.3 迭代优化（4 轮，含量化数据）

#### 第一轮：v0.1→v0.6 — 功能闭环（3 天）

从零搭建，完成"采集→清洗→落库→检索→问答→企微"全链路：

| 里程碑 | 交付物 |
|--------|--------|
| v0.1 | CLI + API + 跨平台抓取 + 基础 RAG |
| v0.2 | Web 仪表盘 + 企微"转发即入库" |
| v0.3 | Wiki 编译引擎 + 知识图谱可视化 |
| v0.4 | 多模态入库（PDF + 图片 OCR + 音频转录） |
| v0.5 | AI 智能助手（多轮对话 + 意图识别 + SSE 流式） |
| v0.6 | 图文穿插 OCR + 长文分段清洗 + 小红书 MCP |

#### 第二轮：v0.8 — Wiki 编译质量优化

**问题**：29 篇文章编译出 42 个 Wiki 页面，51% 重复；17/21 页面被塞进同一个分类。

**改动**：
- 编译前 Embedding 语义去重门控
- 分类从硬编码关键词规则 → LLM 语义分类（预定义 9 分类体系）
- 前端目录树改为读 taxonomy 语义分类

**数据结果**：

| 指标 | 改前 | 改后 |
|------|------|------|
| 重复页面率 | 51% | **0%** |
| 分类数 | 2 个（81% 塞一个桶） | **9 个语义分类** |
| 页面/文章比 | 1.44 | **0.93** |

#### 第三轮：v0.9 — 检索质量 P0 优化

**问题**：Recall@5 只有 41%，一半问题找不到正确文档。根因：用英文 Embedding 模型处理中文内容。

**改动**：
- Embedding 从 `all-MiniLM-L6-v2`(英文/384维) → `BAAI/bge-small-zh-v1.5`(中文/512维)
- 搭建四维度评测框架（忠实度/上下文/答案/延迟分段），60 条用例

**数据结果**：

| 指标 | v0.8 | v0.9 | 提升 |
|------|------|------|------|
| **Recall@5** | 41.3% | **78.3%** | **+37pp** |
| **Hit Rate** | 50.0% | **87.0%** | **+37pp** |
| **MRR** | 0.308 | **0.623** | **+0.315** |

#### 第四轮：v0.9.3 — LLM 切换 + Prompt 强化 + 精排

**问题**：4 条幻觉回答（忠实度 76%）；答案生成延迟 9.3s 占总延迟 65%。

**改动**：
- LLM 从 Kimi 切换到 DeepSeek（响应更快）
- RAG Prompt 加 6 条铁律（无文档不回答/不编造/抗诱导/简洁切题）
- Cross-Encoder Reranker 精排（`bge-reranker-v2-m3`）
- 8 条反事实诱导测试用例

**数据结果**：

| 指标 | v0.9 | v0.9.3 | 变化 |
|------|------|--------|------|
| **Recall@5** | 78.3% | **81.9%** | +3.6pp |
| **Hit Rate** | 87.0% | **91.3%** | +4.3pp |
| **MRR** | 0.623 | **0.659** | +0.036 |
| 否定测试 | — | **100%** | 完美 |
| 零错误运行 | 2 errors | **0 errors** | ✅ |

### 2.5 个人角色与收获

**角色**：独立完成全流程——需求定义 → 架构设计 → 全栈开发 → 部署运维 → 效果评测 → 迭代优化

**收获**：

1. **RAG 系统设计**：深入理解了 Embedding 选型（英文 vs 中文模型对检索质量的决定性影响）、混合检索策略（BM25 + 向量 + RRF + Reranker 四级漏斗）、以及 Prompt Engineering 对生成忠实度的控制

3. **端到端产品思维**：从"微信转发"这个极简入口出发，串联起采集→清洗→编译→检索→问答→推送的完整闭环，月成本控制在 ¥17
