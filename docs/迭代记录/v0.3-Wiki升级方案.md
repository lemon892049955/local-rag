# Local RAG v0.3 — Wiki 编译模式升级规划

> 从"文章仓库 + 检索"进化为"LLM 持续编译的知识 Wiki"
> 参考：[Karpathy LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

**版本**：v0.3.0
**规划日期**：2026-04-08
**修订**：2026-04-08 (融入架构安全性审查的 5 项修正)

---

## 一、升级核心理念

### 现状 (v0.2)

```
URL → 抓取 → LLM 清洗 → 存为独立 .md 文件 → 向量索引
提问 → 从文章片段中检索 → 拼装答案
```

每篇文章是一座孤岛。知识没有积累、没有交叉引用、没有综合。问同一个主题的不同问题，每次都在重新发现知识。

### 目标 (v0.3)

```
URL → 抓取 → LLM 清洗 → 存为原始文章(不可变)
                        ↓
                  编译任务入队 (FIFO)          ← 🔧 修正1: 单线程队列
                        ↓
                   LLM 编译引擎 (串行执行)
                        ↓
              读取现有 Wiki → 制定编译计划
                        ↓
         Append-only 更新页面 → 代码生成索引    ← 🔧 修正2+3: 非破坏性 + 代码生成
                        ↓
提问 → Wiki(Top-2) + Data(Top-3) 并行召回      ← 🔧 修正5: 并行多路
                        ↓
         LLM 综合答案 → 异步后台回填判定        ← 🔧 修正4: 异步回填
```

**核心转变**：入库不再只是"存一篇文章"，而是**触发一次知识编译** — LLM 阅读新文章 + 现有 Wiki，决定创建/更新哪些 Wiki 页面，让知识网络持续生长。

## 二、架构安全性设计 (5 项关键修正)

> 以下 5 项是对初版规划的架构审查，识别出的致命隐患和对应修正方案。

### 🔧 修正 1：单线程编译队列 — 防并发冲突

**隐患**：用户连续转发 3 篇文章，3 个编译进程并行启动，同时读写 `_index.md` 和同一个 Wiki 页面，导致文件冲突或数据覆盖。

**修正**：入库 (Ingest) 和编译 (Compile) 严格解耦，编译走**单线程 FIFO 队列**。

```
入库流程 (可高并发)          编译队列 (严格串行)
┌──────────────────┐       ┌──────────────────────┐
│ 文章A → data/A.md │──┐    │ Queue: [A, B, C]     │
│ 文章B → data/B.md │──┼──→ │                      │
│ 文章C → data/C.md │──┘    │ Worker: 正在编译 A... │
└──────────────────┘       │         → 编译 B...   │
                           │         → 编译 C...   │
                           └──────────────────────┘
```

**实现方案**：`asyncio.Queue` + 单个后台消费者协程。入库完成后只做 `queue.put(article_path)`，编译 Worker 逐个取出串行处理。

```python
# wiki/compile_queue.py

import asyncio
import logging
from pathlib import Path

logger = logging.getLogger(__name__)

_compile_queue: asyncio.Queue = None
_worker_task: asyncio.Task = None


def get_queue() -> asyncio.Queue:
    global _compile_queue
    if _compile_queue is None:
        _compile_queue = asyncio.Queue()
    return _compile_queue


async def enqueue_compile(article_path: Path):
    """入库完成后调用：将文章加入编译队列"""
    queue = get_queue()
    await queue.put(article_path)
    logger.info(f"编译任务入队: {article_path.name} (队列长度: {queue.qsize()})")


async def compile_worker():
    """单线程编译消费者 — 严格串行执行"""
    from wiki.compiler import WikiCompiler
    compiler = WikiCompiler()
    queue = get_queue()

    while True:
        article_path = await queue.get()
        try:
            logger.info(f"开始编译: {article_path.name}")
            result = await compiler.compile(article_path)
            logger.info(f"编译完成: {article_path.name}, 影响 {len(result.affected_pages)} 个页面")
        except Exception as e:
            logger.error(f"编译失败: {article_path.name} - {e}")
        finally:
            queue.task_done()


async def start_compile_worker():
    """在应用启动时调用，启动编译 Worker"""
    global _worker_task
    if _worker_task is None or _worker_task.done():
        _worker_task = asyncio.create_task(compile_worker())
        logger.info("Wiki 编译 Worker 已启动")
```

### 🔧 修正 2：Append-only 非破坏性更新 — 防知识磨损

**隐患**：Wiki 页面 `产品方法论.md` 经 10 篇文章积累有 3000 字干货。新文章入库后让 LLM "更新页面"，LLM 偷懒过度总结，把 3000 字压缩成 1500 字。更新越多，知识越空洞。

**修正**：`_schema.md` 中严格约束 **Append-only** 更新原则。LLM 不允许重写已有段落，只能：

1. **追加新章节**：在页面底部的 `## 新增洞察` 区域追加内容
2. **追加交叉引用**：在 `## 相关概念` 区域补充链接
3. **标记矛盾**：用 `> ⚠️ 矛盾` blockquote 标注新旧信息冲突，**不删除旧信息**

```markdown
# 产品方法论  ← 已有内容不动

## 核心原则            ← 已有段落，LLM 不允许修改
...3000 字干货...

## 相关概念            ← 可追加新链接
- [[Vibe Coding]]
- [[AI应用趋势]]      ← 新增

## 新增洞察            ← LLM 只在这里追加
### 2026-04-08 | 来源: 腾讯产品经理晋升的共同点
影响力半径是腾讯产品经理晋升的核心评估维度...
[来源: data/260407_bNodk8cE_腾讯产品经理晋升的共同点.md]

> ⚠️ 矛盾: 本文强调"影响力半径"，而此前的来源更强调"方法论框架"。两者可能互补而非矛盾。
```

**编译 Prompt 中的硬约束**：

```
## 更新规则 (CRITICAL)

当更新已有 Wiki 页面时，你 **绝对不允许** 做以下事情：
1. 删除或改写已有的任何段落
2. 对已有内容进行压缩、总结、合并
3. 调整已有内容的结构或顺序

你 **只允许** 做：
1. 在 "## 新增洞察" 区域追加新的三级标题和内容
2. 在 "## 相关概念" 区域追加新的交叉引用链接
3. 用 "> ⚠️ 矛盾:" blockquote 标记新旧信息的冲突（保留旧信息不动）
```

### 🔧 修正 3：代码生成索引/日志 — 防 LLM 幻觉

**隐患**：让 LLM 维护 `_index.md` 目录，当 Wiki 达到 100+ 页面时，LLM 容易幻觉（编造不存在的链接、遗漏刚创建的页面），且浪费大量 Token。

**修正**：**让代码的归代码，LLM 的归 LLM。**

- `_index.md`：**Python 脚本自动生成**。每次编译完成后，遍历 `wiki/` 目录所有 `.md` 文件，读取 YAML front-matter，自动生成目录。100% 准确，毫秒级，零 Token。
- `_log.md`：**Python 代码追加写入** (`file.write(mode='a')`)。不需要 LLM 参与，编译引擎执行完后直接 append 一条结构化日志。
- LLM 编译时的上下文：不再读 `_index.md` 全文，改为读取 **Python 生成的轻量摘要**（页面名列表 + 一句话描述）。

```python
# wiki/index_builder.py

from pathlib import Path
from datetime import datetime
import yaml

def rebuild_index(wiki_dir: Path) -> str:
    """遍历 wiki/ 目录，自动生成 _index.md — 零 LLM 调用"""
    sections = {"topics": [], "entities": [], "insights": []}

    for subdir in ["topics", "entities", "insights"]:
        dir_path = wiki_dir / subdir
        if not dir_path.exists():
            continue
        for md_file in sorted(dir_path.glob("*.md")):
            meta = _read_frontmatter(md_file)
            if meta:
                sections[subdir].append({
                    "file": f"{subdir}/{md_file.name}",
                    "title": meta.get("title", md_file.stem),
                    "summary": meta.get("summary", ""),
                    "sources_count": len(meta.get("sources", [])),
                    "updated_at": meta.get("updated_at", ""),
                })

    # 生成 Markdown
    lines = ["# Wiki 索引\n", f"> 自动生成于 {datetime.now().strftime('%Y-%m-%d %H:%M')}\n"]

    for section_name, label in [("topics", "主题"), ("entities", "实体"), ("insights", "洞察")]:
        items = sections[section_name]
        lines.append(f"\n## {label} ({len(items)})\n")
        if items:
            lines.append("| 页面 | 摘要 | 来源数 | 更新时间 |")
            lines.append("|------|------|--------|---------|")
            for item in items:
                lines.append(f"| [[{item['title']}]]({item['file']}) | {item['summary'][:50]} | {item['sources_count']} | {item['updated_at']} |")
        else:
            lines.append("_(暂无)_")

    content = "\n".join(lines) + "\n"
    (wiki_dir / "_index.md").write_text(content, encoding="utf-8")
    return content


def append_log(wiki_dir: Path, action: str, source: str, details: list[str]):
    """追加编译日志 — 纯代码写入，不经过 LLM"""
    log_path = wiki_dir / "_log.md"
    now = datetime.now()
    date_str = now.strftime("%Y-%m-%d")
    time_str = now.strftime("%H:%M")

    entry = f"- **{time_str} {action}** | 来源: `{source}`\n"
    for detail in details:
        entry += f"  - {detail}\n"

    # 读取现有内容，在对应日期下追加
    if log_path.exists():
        content = log_path.read_text(encoding="utf-8")
        date_header = f"## {date_str}"
        if date_header in content:
            # 在日期标题后追加
            content = content.replace(date_header, f"{date_header}\n\n{entry}", 1)
        else:
            # 新日期，插入到标题后
            content = content.replace("# Wiki 操作日志\n", f"# Wiki 操作日志\n\n{date_header}\n\n{entry}", 1)
    else:
        content = f"# Wiki 操作日志\n\n## {date_str}\n\n{entry}"

    log_path.write_text(content, encoding="utf-8")


def build_lightweight_summary(wiki_dir: Path) -> str:
    """为 LLM 编译决策生成轻量摘要（替代读取完整 _index.md）"""
    lines = ["当前 Wiki 已有页面:\n"]
    for subdir in ["topics", "entities", "insights"]:
        dir_path = wiki_dir / subdir
        if not dir_path.exists():
            continue
        for md_file in sorted(dir_path.glob("*.md")):
            meta = _read_frontmatter(md_file)
            title = meta.get("title", md_file.stem) if meta else md_file.stem
            summary = meta.get("summary", "")[:60] if meta else ""
            lines.append(f"- [{subdir}] {title}: {summary}")
    return "\n".join(lines)


def _read_frontmatter(filepath: Path) -> dict:
    try:
        content = filepath.read_text(encoding="utf-8")
        if not content.startswith("---"):
            return None
        parts = content.split("---", 2)
        if len(parts) < 3:
            return None
        return yaml.safe_load(parts[1]) or {}
    except Exception:
        return None
```

### 🔧 修正 4：异步回填 + 价值门槛 — 防查询延迟

**隐患**：用户在企微问问题，检索 + LLM 回答需要 5 秒。如果加上同步回填（判断是否建洞察页 + 写文件 + 更新索引），用户可能等 30 秒以上。且临时性问题会制造大量垃圾洞察页。

**修正**：查询链路 (Query) 必须**绝对轻量**。回填完全异步化，且设严格价值门槛。

```
用户提问
    ↓
Step 1: 查询改写 + 并行多路召回 (见修正5)
    ↓
Step 2: LLM 生成答案
    ↓
Step 3: 立即返回答案给用户          ← 到这里为止，用户体验结束
    ↓
Step 4: (异步后台) 回填价值评估      ← 不阻塞用户
    → 条件必须同时满足:
      a. 答案引用了 ≥ 3 个不同来源
      b. 答案长度 > 500 字
      c. Wiki 中不存在同名/近似主题的洞察页
    → 满足 → 入编译队列 (复用修正1的队列)
    → 不满足 → 丢弃，不回填
```

**关键**：回填任务走编译队列（修正 1），不会产生新的并发问题。

### 🔧 修正 5：并行多路召回 — 防信息缺失

**隐患**：基于阈值的条件 Fallback 不可靠。RRF 分数只代表语义相关性，不代表信息详尽度。Wiki 主题页可能语义匹配高但缺少具体细节（如代码命令），导致回答空洞。

**修正**：抛弃条件 Fallback，改为**每次查询都并行召回两路**。

```
用户提问
    ↓
查询改写 (复用现有逻辑)
    ↓
┌──────────────────────┬──────────────────────┐
│ Wiki 路 (宏观上下文)   │ Data 路 (微观事实)     │
│ 向量搜索 wiki/        │ 向量搜索 data/         │
│ BM25 搜索 wiki/       │ BM25 搜索 data/        │
│ RRF 融合 → Top-2      │ RRF 融合 → Top-3       │
└──────────┬───────────┴──────────┬───────────┘
           │                      │
           ▼                      ▼
      ┌─────────────────────────────┐
      │ 合并为 Context 喂 LLM       │
      │ Wiki 页面提供"树干"(结构)    │
      │ Data 切片提供"树叶"(细节)    │
      └──────────┬──────────────────┘
                 ↓
           LLM 综合答案
```

**实现变化**：`hybrid_searcher.py` 中维护两个独立的索引集合：
- `wiki_collection`：Wiki 页面的向量索引
- `data_collection`：原始文章的向量索引（现有）

BM25 也分两套。每次查询并行跑两路，合并后一起喂 LLM。

**Context 组装模板**：

```
---以下是知识 Wiki 的相关页面（提供结构化上下文）---
{wiki_context}

---以下是原始文章的相关片段（提供具体事实和细节）---
{data_context}
```

### 双层数据分离

```

│   │

### 层级职责 (修正后)

| 层 | 内容 | 谁写 | 可变性 | 并发安全 |
|----|------|------|--------|---------|
| `data/` | 原始文章 | 入库流程 | **不可变** | 可并发写入（文件名唯一） |
| `wiki/` 页面 | 主题/实体/洞察页 | **LLM 编译引擎** (Append-only) | 持续追加 | 🔧 单线程队列保护 |

| `wiki/_log.md` | 操作日志 | 🔧 **Python 代码** | 追加写入 | 队列内串行，无冲突 |
| `vectordb/` | 向量索引 | 索引器 | 编译后增量更新 | 队列内串行，无冲突 |

## 四、核心模块设计

### 4.1 编译队列 (`wiki/compile_queue.py`) — 🔧 新增

见修正 1 的代码设计。核心是 `asyncio.Queue` + 单消费者协程。

- `enqueue_compile(article_path)` — 入库完成后调用
- `compile_worker()` — 串行消费，保证同一时间只有一个编译在执行
- `start_compile_worker()` — FastAPI 启动时调用

### 4.2 Wiki 编译引擎 (`wiki/compiler.py`) — 核心模块

**编译流程** (修正后):

```
输入: 新文章路径 (data/xxx.md)

Step 1: 读取新文章内容
   → 解析 front-matter + 正文

Step 2: 获取 Wiki 轻量摘要
   → 🔧 调用 index_builder.build_lightweight_summary()
   → 返回页面名列表 + 一句话描述 (非完整 _index.md)

Step 3: LLM 制定编译计划
   → 输入: 新文章内容 + Wiki 轻量摘要
   → 输出 JSON: new_pages[] + update_pages[] + cross_references[]
   → 🔧 update_pages 的 action 限定为: "append" | "flag_conflict" (无 "revise")

Step 4: 执行编译
   → 新建页面: LLM 生成完整内容
   → 🔧 更新页面: 读取已有内容 → LLM 只生成追加段落 → Python 拼接到 "## 新增洞察" 区域
   → 交叉引用: Python 代码在 "## 相关概念" 区域追加链接

Step 5: 🔧 代码更新索引 + 日志
   → index_builder.rebuild_index()   — Python 自动生成
   → index_builder.append_log()      — Python 追加写入

Step 6: 增量向量索引
   → 对本次创建/更新的 Wiki 页面重新索引到 wiki_collection

输出: CompileResult(affected_pages, new_pages, updated_pages)
```

### 4.3 模式文件 (`wiki/_schema.md`) — 编译规则

```markdown
# Wiki 编译模式文件

## 角色
你是知识 Wiki 编译器。当用户添加新文章时，你负责将新知识编译到 Wiki 中。

## 页面类型
- **主题页 (topics/)**: 围绕一个概念/方法论/技术，综合多个来源的知识
- **实体页 (entities/)**: 围绕一个人物/公司/产品，记录相关事实
- **洞察页 (insights/)**: 跨主题的对比、综合分析（仅由查询回填生成）

## 创建新页面的条件
- 新文章涉及一个 Wiki 中尚不存在的核心概念/实体
- 该概念/实体在文章中有足够的信息量（不是一笔带过）
- 不要为过于宽泛的概念建页（如"技术""互联网"）

## 🚫 更新规则 (CRITICAL — 非破坏性原则)

当更新已有 Wiki 页面时，你 **绝对不允许** 做以下事情：
1. 删除或改写已有的任何段落
2. 对已有内容进行压缩、总结、合并
3. 调整已有内容的结构或顺序
4. 用 "revise" 类型的操作

你 **只允许** 做以下操作：
1. 在 "## 新增洞察" 区域追加新的三级标题和内容（格式: ### YYYY-MM-DD | 来源: xxx）
2. 在 "## 相关概念" 区域追加新的 [[交叉引用]] 链接
3. 用 "> ⚠️ 矛盾:" blockquote 标记新旧信息的冲突（保留旧信息不动）

## 页面模板

### 新建主题页模板
​```
---
type: topic
title: {标题}

### 4.4 检索层改造 — 🔧 并行多路召回

```
用户提问
    ↓
Step 1: 查询改写 (复用现有 HybridSearcher 的 _rewrite_query)
    ↓
Step 2: 并行双路召回 (每次都跑，不做条件判断)
    ┌─────────────────────────────┐
    │ Wiki 路 (宏观结构)            │
    │  向量搜索 wiki_collection     │
    │  BM25 搜索 wiki_bm25_index   │
    │  RRF 融合 → Top-2 Wiki 页面  │
    ├─────────────────────────────┤
    │ Data 路 (微观细节)            │
    │  向量搜索 data_collection     │
    │  BM25 搜索 data_bm25_index   │
    │  RRF 融合 → Top-3 Data 切片  │
    └──────────────┬──────────────┘
                   ↓
Step 3: 组装双层 Context
    Wiki 页面提供 "树干" (结构化知识、交叉引用)
    Data 切片提供 "树叶" (具体事实、代码、数据)
                   ↓
Step 4: LLM 综合答案 → 立即返回给用户
                   ↓
Step 5: (异步后台) 回填价值评估
    条件: 引用 ≥3 来源 AND 答案 > 500 字 AND Wiki 无近似页
    → 满足 → enqueue_compile() 入编译队列
    → 不满足 → 丢弃
```

### 4.5 健康检查 (`wiki/inspector.py`) — 简单版

**检查项** (纯 Python 实现，无 LLM 调用):
- **孤立页面**：扫描所有 `[[]]` 引用，找出零入链的页面
- **缺失页面**：被引用但文件不存在的页面
- **过时页面**：超过 30 天未更新且来源数 ≤ 1
- **来源统计**：每个 Wiki 页面关联的原始来源数量

**触发方式**: `POST /wiki/inspect` 或 `python cli.py wiki-inspect`

## 六、关键设计决策记录

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | 原始来源与 Wiki 的关系 | **双层分离** | data/ 不可变保证数据安全，wiki/ 由 LLM 编译 |
| D2 | Wiki 页面粒度 | **LLM 自主决定** | 通过 `_schema.md` 约束质量和门槛 |
| D3 | 编译交互方式 | **全自动 + 后续加审核** | 先跑起来积累数据 |
| D4 | 检索策略 | 🔧 **并行多路召回** | Wiki(Top-2) + Data(Top-3) 一起喂 LLM，不做阈值判断 |
| D5 | 索引和日志 | 🔧 **代码生成** | _index.md 由 Python 脚本生成，_log.md 由代码追加写入 |
| D6 | 健康检查 | **简单版先行** | 纯 Python 实现，不依赖 LLM |
| D7 | 🔧 并发控制 | **单线程编译队列** | asyncio.Queue + 单消费者，杜绝并发冲突 |
| D8 | 🔧 更新策略 | **Append-only** | LLM 只追加不改写，防止知识磨损 |
| D9 | 🔧 查询回填 | **异步 + 价值门槛** | 不阻塞查询响应，≥3来源 + >500字 才回填 |
