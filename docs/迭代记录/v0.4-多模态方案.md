# Local RAG v0.4 — 多模态转文字能力规划

> 让知识库不止吃链接，还能吃图片、PDF、音频
> 参考架构：输入源 → 路由分发 → 专用解析器 → 标准化池 → 编译引擎

**版本**：v0.4.0
**规划日期**：2026-04-08

---

## 一、架构总览

```
输入源              路由分发          专用解析器          标准化池        编译引擎
┌──────────┐      ┌──────────┐     ┌──────────┐      ┌──────────┐   ┌──────────┐
│ 网页链接  │─┐    │          │──→  │ SCRAPER  │──┐   │          │   │          │
│ 小红书图文│─┼──→ │DISPATCHER│──→  │VISION OCR│──┼─→ │ MARKDOWN │─→ │LLM ENGINE│
│ PDF 报告 │─┤    │          │──→  │PDF PARSER│──┤   │  (SSOT)  │   │ (Wiki    │
│ 播客音频  │─┘    │          │──→  │ WHISPER  │──┘   │          │   │  编译)   │
└──────────┘      └──────────┘     └──────────┘      └──────────┘   └──────────┘
```

**核心设计原则**：所有输入源经过专用解析器后，统一输出为**纯文本/Markdown**，流入现有的 LLM 清洗 → Markdown 落库 → Wiki 编译管线。解析器只做"转文字"，不做清洗。

## 一.5、服务器约束与成本分析

### 当前服务器配置

| 项目 | 规格 |
|------|------|
| 服务器 | 腾讯云轻量应用服务器 (上海) |
| CPU | 2 核 |
| 内存 | 4 GB (Docker 容器约占 2.5 GB) |
| 磁盘 | 60 GB SSD |
| 镜像体积 | local-rag:fixed 约 5.85 GB |
| 无 GPU | ❌ |

### 约束与决策原则

1. **内存极其紧张** — 4GB 总内存，Docker 已占 2.5GB，剩余 ~1.5GB 可用。**绝对不能**加载大模型到本地内存
2. **无 GPU** — 排除所有需要 GPU 的本地方案（faster-whisper、llama-vision 等）
3. **镜像体积已达 5.85GB** — 每多装一个大依赖都在增加构建时间和磁盘占用。**优先选最小依赖**
4. **API 调用优先** — 计算密集型任务（OCR、转录）全部走云端 API，服务器只做调度和存储

### 各方案成本对比

#### PDF 解析

| 方案 | 包体积 | 内存占用 | 运行成本 | 选择 |
|------|--------|---------|---------|------|
| **pymupdf** | ~15MB | ~20MB | 免费 | **✅ 首选** — 纯 CPU，极轻量 |
| pdfplumber | ~8MB | ~15MB | 免费 | 备选 — 更小但功能弱 |
| pdfminer.six | ~5MB | ~10MB | 免费 | 备选 — 最小但结构解析差 |
| Unstructured | ~500MB+ | ~1GB | 免费 | ❌ 太重 |

**决定**：**pymupdf**。15MB 包体积，CPU 就能跑，文本提取 + 页面渲染都支持。

#### 图片 OCR

| 方案 | 包体积 | 内存占用 | 每张成本 | 选择 |
|------|--------|---------|---------|------|
| **Kimi Vision API** | 0 (复用 openai SDK) | 0 | ~¥0.01/张 | **✅ 首选** — 零新依赖 |
| OpenAI GPT-4o | 0 (复用 openai SDK) | 0 | ~$0.01/张 | 备选 |
| Tesseract OCR | ~30MB + 语言包 ~50MB | ~200MB | 免费 | ❌ 中文效果差，体积大 |
| PaddleOCR | ~800MB+ | ~1GB+ | 免费 | ❌ 太重，需 GPU |
| llama-3.2-vision (Ollama) | ~4GB | ~4GB | 免费 | ❌ 内存不够 |

**决定**：**Kimi Vision API**（`moonshot-v1-vision`）。零新依赖，零内存开销，复用已有的 OpenAI SDK 和 API Key。每张图约 ¥0.01，一天 OCR 10 张也就 ¥0.1。

#### 音频转录

| 方案 | 包体积 | 内存占用 | 每分钟成本 | 选择 |
|------|--------|---------|-----------|------|
| **OpenAI Whisper API** | 0 (复用 openai SDK) | 0 | ~$0.006/分钟 | **✅ 首选** — 零新依赖 |
| faster-whisper (tiny) | ~200MB | ~500MB | 免费 | ❌ 需 GPU 才快 |
| faster-whisper (base) | ~400MB | ~1GB | 免费 | ❌ 内存不够 |
| Whisper.cpp (tiny) | ~80MB 二进制 | ~200MB | 免费 | 备选 — 纯 CPU 但慢 |
| 腾讯云 ASR API | 0 | 0 | ~¥0.007/15秒 | 备选 — 中文可能更好 |

**决定**：**OpenAI Whisper API**。零新依赖，零内存开销。一期播客 60 分钟约 $0.36 (¥2.5)，个人使用完全可接受。后续如果成本敏感，可以换腾讯云 ASR（中文效果可能更好，价格接近）。

## 三、四种新输入源详细设计

### 3.1 小红书图文 → Vision OCR

**场景**：小红书的图文笔记，核心内容在图片里（长图、信息图、PPT 截图等），纯文本抓取拿不到。

**解析器**：`ingestion/vision_ocr.py`

**方案**：Kimi Vision API（`moonshot-v1-vision`）— 零新依赖，零内存开销，复用已有 OpenAI SDK + API Key

```
输入: 图片 URL 列表 或 本地图片文件
      ↓
Step 1: 下载图片 (requests)，压缩到 ≤1MB (PIL/Pillow 已在 sentence-transformers 依赖中)
Step 2: Base64 编码图片
Step 3: 调用 Kimi Vision API (OpenAI 兼容格式，messages 中传 image_url)
        Prompt: "请提取这张图片中的所有文字内容，保留原始结构，输出为纯文本"
Step 4: 多张图片结果拼接
      ↓
输出: RawContent(content=拼接文本, source_platform="image")
```

**成本**：~¥0.01/张 | **包体积增量**：0 | **内存增量**：0

**触发方式**：
- 企微转发小红书链接 → 现有抓取器抓到文字内容过短 → 自动检测图片 → 调用 Vision OCR 补充
- Web UI 直接上传图片 → 新增 `/ingest/image` 端点

---

### 3.2 PDF 报告 → PDF Parser

**场景**：行业报告、论文、白皮书等 PDF 文件。

**解析器**：`ingestion/pdf_parser.py`

**方案**：pymupdf（PyMuPDF）— 15MB 包，纯 CPU，20MB 内存，本地免费运行

```
输入: PDF 文件路径 或 URL
      ↓
Step 1: 下载 PDF (如果是 URL)
Step 2: pymupdf 文本提取
        - 优先提取可选择文本 (非扫描件)
        - 保留标题层级、列表结构
Step 3: 如果文本提取失败/过短 → 降级为 OCR 模式
        - 每页转图片 → Kimi Vision API 提取文字 (复用 vision_ocr.py)
Step 4: 拼接所有页面文本
Step 5: 转录完成后删除原始 PDF 文件（节省磁盘）
      ↓
输出: RawContent(content=全文文本, title=PDF标题, source_platform="pdf")
```

**成本**：文本型 PDF ¥0 | 扫描型 PDF ~¥0.01×页数 | **包体积增量**：~15MB | **内存增量**：~20MB

**依赖**：
- `pymupdf` — 唯一新增包，轻量高效
- 扫描件降级复用 Vision OCR（零额外依赖）

**触发方式**：
- 企微发送 PDF 文件 → 下载 → 解析 → 入库
- Web UI 上传 PDF → 新增 `/ingest/file` 端点
- CLI: `python cli.py ingest-file report.pdf`

### 3.3 播客音频 → Whisper

**场景**：播客节目、语音备忘录、会议录音。

**解析器**：`ingestion/audio_transcriber.py`

**方案**：OpenAI Whisper API — 零新依赖，零内存开销，复用已有 OpenAI SDK

```
输入: 音频文件路径 或 URL (mp3/m4a/wav)
      ↓
Step 1: 下载音频 (如果是 URL)
Step 2: 压缩/分段 (如果 > 25MB，用 pydub 分段，pydub 仅 ~1MB)
Step 3: 调用 OpenAI Whisper API (client.audio.transcriptions.create)
        model="whisper-1", response_format="text"
Step 4: 分段结果拼接
Step 5: 转录完成后删除原始音频文件（节省磁盘）
      ↓
输出: RawContent(content=转录文本, title=文件名, source_platform="audio")
```

**成本**：~$0.006/分钟 (~¥2.5/小时) | **包体积增量**：~1MB (pydub) | **内存增量**：0

> 注：pydub 需要系统安装 ffmpeg。Docker 镜像中需 `apt-get install ffmpeg`（约 +80MB 镜像体积）。
> 如果暂时不需要音频，可以跳过 Phase 3，不装 ffmpeg，镜像零增长。

**触发方式**：
- 企微发送语音/文件 → 下载 → 转录 → 入库
- Web UI 上传音频 → `/ingest/file` 端点
- CLI: `python cli.py ingest-file podcast.mp3`

## 四、路由分发器扩展

现有 `ingestion/router.py` 只处理 URL。需要扩展为统一 Dispatcher，支持 URL + 文件：

```python
class Dispatcher:
    """统一路由分发 — 支持 URL 和文件"""

    def dispatch(self, input) -> str:
        """判断输入类型，返回解析器名称"""
        if is_url(input):
            return detect_source(input)  # 现有逻辑: wechat/xiaohongshu/general
        elif is_pdf(input):
            return "pdf"
        elif is_image(input):
            return "image"
        elif is_audio(input):
            return "audio"
        else:
            return "text"  # 纯文本直接入库
```

**文件类型检测**：基于扩展名 + MIME type
- 图片: `.jpg`, `.jpeg`, `.png`, `.webp`
- PDF: `.pdf`
- 音频: `.mp3`, `.m4a`, `.wav`, `.ogg`, `.flac`

## 九、数据流完整路径

```
用户发一张图片给企微
    ↓
callback.py 识别 image 消息类型
    ↓
下载图片到临时目录
    ↓
Dispatcher: detect_type → "image"
    ↓
vision_ocr.py: Vision LLM 提取文字
    ↓
输出 RawContent(content="图片中的文字...", source_platform="image")
    ↓
===== 以下复用现有管线 =====
    ↓
llm_cleaner.py: LLM 清洗 → CleanedKnowledge
    ↓
markdown_engine.py: 落库 → data/260408_xxx_标题.md
    ↓
indexer.py: 向量索引
    ↓
compile_queue.py: Wiki 编译入队
    ↓
compiler.py: 编译到 Wiki
    ↓
sender.py: 推送"✅ 入库成功"
```

**核心优势**：3 个新解析器只负责"转文字"，后面的清洗→落库→索引→Wiki编译全部复用，零改动。
