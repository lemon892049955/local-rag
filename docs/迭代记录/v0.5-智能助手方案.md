# Local RAG v0.5 — 智能助手 + 主动推送规划

> 从"被动检索"进化为"主动服务"的知识助手
> 核心形态：右下角悬浮球 AI 助手，集对话、推送、入库于一体

**版本**：v0.5.0
**规划日期**：2026-04-08

---

## 一、核心理念

### 现状 (v0.4)

用户必须主动操作才能获取知识：点入库、点搜索、点图谱。知识库是被动的。

### 目标 (v0.5)

知识库变为**主动服务的智能助手**：
- 随时可对话（悬浮窗，不离开当前页面）
- 主动推送（每周摘要、知识回顾、关联推荐）
- 一个入口完成所有操作（对话即入库、对话即搜索）

## 二、悬浮窗助手设计

### 2.1 交互形态

```
状态1: 收起 — 右下角悬浮球
┌──────────────────────────┐
│       任意页面内容         │
│                          │
│                     🧠   │  ← 悬浮球，带未读红点
└──────────────────────────┘

状态2: 展开 — 对话窗 (400x600)
┌──────────────────────────┐
│       任意页面内容         │
│              ┌───────────────────┐
│              │ 🧠 知识助手         │
│              │ ─────────────────  │
│              │ 📬 每周摘要 (新)    │ ← 推送消息卡片
│              │ ─────────────────  │
│              │ 你: vibe coding    │ ← 用户消息
│              │    怎么做？         │
│              │                   │
│              │ AI: 根据 Wiki      │ ← AI 回答 (Markdown)
│              │  《Vibe Coding     │
│              │   工作流程详解》    │ ← 可点击跳转
│              │  核心步骤是...      │
│              │  [来源: ...]       │
│              │ ─────────────────  │
│              │ 💡 你可能想看:      │ ← 关联推荐
│              │  · AI产品经理的挑战  │
│              │ ─────────────────  │
│              │ [输入消息... 📎 🎤] │ ← 输入框+附件+语音
│              └───────────────────┘
└──────────────────────────┘
```

### 2.2 功能模块

| 模块 | 说明 |
|------|------|
| **对话** | 多轮对话，自动调用 RAG 检索，回答带 Wiki/文章引用链接 |
| **入库** | 粘贴链接自动识别入库，拖入文件入库，无需切页面 |
| **推送流** | 时间线形式展示系统推送（每周摘要、知识回顾、编译通知） |
| **快捷操作** | "总结本周入库" / "推荐相关知识" / "Wiki 健康检查" 快捷按钮 |

### 2.3 对话能力设计

```
用户输入 → 意图识别
    ├── 包含 URL → 自动入库（调用 /ingest）
    ├── 包含 "总结" "本周" "摘要" → 生成知识摘要
    ├── 包含 "推荐" "相关" → 关联推荐
    ├── 包含 "?" 或问句 → RAG 检索（调用 /search）
    └── 其他 → 自由对话（带知识库上下文）
```

**多轮上下文**：前端维护 `messages[]` 数组，每次请求带上最近 5 轮对话历史，后端拼接到 LLM Prompt 中。

### 2.4 后端新增 API

```
POST /api/chat
{
  "message": "vibe coding 怎么做？",
  "history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}

Response:
{
  "reply": "根据 Wiki《Vibe Coding 工作流程详解》...",
  "sources": [...],
  "action": null,           // "ingest" | "summary" | null
  "suggestions": ["查看 Wiki 详情", "相关: AI产品经理"],
  "notifications": [...]    // 待推送的通知
}
```

## 三、主动推送系统

### 3.1 推送类型

| 类型 | 触发 | 内容 | 频率 |
|------|------|------|------|
| **每周知识摘要** | 每周日 20:00 定时 | LLM 读取本周入库文章 + Wiki 变化，生成结构化周报 | 每周 |
| **知识回顾** | 每天 10:00 定时 | "7 天前你收藏了《XXX》，核心观点是..." | 每天 |
| **关联推荐** | 入库新文章后 | "你刚入库的文章和《XXX》有关联，要不要看看？" | 事件驱动 |
| **编译通知** | Wiki 编译完成后 | "📝 Wiki 编译完成，新建/更新了 N 个页面" | 事件驱动 |
| **健康提醒** | 每月 1 号 | "知识库有 N 个孤立页面，建议检查" | 每月 |

### 3.2 推送渠道

| 渠道 | 场景 | 实现 |
|------|------|------|
| **悬浮窗助手** (主渠道) | 用户打开 Web 时看到 | 前端轮询 `/api/notifications` |
| **企微消息** (辅助) | 用户不在 Web 时 | 复用现有 `sender.py` |

### 3.3 推送存储

```
notifications/             # 推送消息存储 (Markdown 文件)
├── 20260414_weekly.md     # 每周摘要
├── 20260408_recall.md     # 知识回顾
└── ...
```

后端 API：
```
GET  /api/notifications           # 获取未读推送列表
POST /api/notifications/{id}/read # 标记已读
POST /api/notifications/generate  # 手动触发生成（调试用）
```

## 四、每周知识摘要详细设计

### 触发

- 定时：每周日 20:00（后台定时任务）
- 手动：助手对话中说"总结本周"

### 生成流程

```
Step 1: 收集本周数据
  → 本周新入库的文章列表 (data/ 按时间筛选)
  → 本周 Wiki 编译日志 (_log.md 按日期筛选)
  → 本周 Wiki 新增/更新的页面

Step 2: LLM 生成周报
  Prompt: "你是知识库助手。以下是用户本周的知识库活动，请生成一份简洁的周报..."
  输出: Markdown 格式周报

Step 3: 存储 + 推送
  → 保存到 notifications/
  → 悬浮窗显示红点
  → 企微推送摘要版
```

### 周报模板

```markdown
# 📊 知识周报 | 2026-04-07 ~ 04-13

## 本周入库
共入库 **5** 篇新知识，涵盖 AI 技术、产品方法论 两大主题。

### 重点文章
1. **Vibe Coding 工作流程详解** — 通过重组产品构建流程实现高效转化
2. **AI 产品经理的挑战** — 技术落地难与商业化困境

## Wiki 变化
- 新建 3 个主题页：Vibe Coding、AI 就业趋势、短剧角色设计
- 更新 2 个已有页面：AI 技术梳理、产品方法论
- 新增 8 条交叉引用

## 知识洞察
本周入库内容集中在 **AI 应用落地** 方向，建议关注...

## 下周建议
- 「AI 产品经理」主题已有 3 篇来源，可以深入阅读
- 「Vibe Coding」和「产品方法论」有关联，建议探索交叉领域
```

## 五、知识回顾设计

### 触发

每天 10:00，从已入库文章中选取 N 天前的 1-2 篇推送回顾。

### 选择策略

```python
def select_recall_articles(days_ago=7):
    """选择 N 天前入库的文章"""
    target_date = today - timedelta(days=days_ago)
    candidates = [f for f in data_dir.glob("*.md") 
                  if f.stat().st_mtime matches target_date]
    # 优先选标签多、Wiki 关联多的文章
    return sorted(candidates, key=quality_score, reverse=True)[:2]
```

### 推送内容

```
🔔 知识回顾

7 天前你收藏了:
📄 **Vibe Coding 工作流程详解**
> 核心观点: 通过重组产品构建流程，实现从想法到产品落地的高效转化

💡 这篇文章和 Wiki 中的「产品方法论」「迭代开发」有关联。
```

## 六、关联推荐设计

### 触发

每次入库新文章 + Wiki 编译完成后。

### 推荐逻辑

```
新文章入库 → Wiki 编译完成
    ↓
读取新文章的标签和 Wiki 关联
    ↓
在已有文章/Wiki 中找 标签交集 ≥ 2 或 共享 Wiki 页面 的文章
    ↓
生成推荐: "你刚入库的文章和《XXX》有 N 个共同主题"
    ↓
推送到悬浮窗 + 企微
```

## 七、定时任务框架

### 方案

用 `asyncio` 后台定时任务（不引入 Celery 等重依赖）：

```python
# scheduler.py
async def scheduler():
    """后台定时任务调度器"""
    while True:
        now = datetime.now()
        
        # 每天 10:00 — 知识回顾
        if now.hour == 10 and now.minute == 0:
            await generate_recall()
        
        # 每周日 20:00 — 每周摘要
        if now.weekday() == 6 and now.hour == 20 and now.minute == 0:
            await generate_weekly_summary()
        
        # 每月 1 号 9:00 — 健康提醒
        if now.day == 1 and now.hour == 9 and now.minute == 0:
            await generate_health_reminder()
        
        await asyncio.sleep(60)  # 每分钟检查一次
```

在 FastAPI startup 中启动：`asyncio.create_task(scheduler())`

### Phase 2: 推送系统 + 通知存储

| 任务 | 说明 |
|------|------|

| 任务 | 说明 |

### Phase 4: 关联推荐 + 快捷操作

| 任务 | 说明 |
