# v0.6 入库场景优化 — 技术方案

## 一、版本目标

解决当前入库链路中**核心内容丢失**（图片信息、视频页面描述）和**链路 Bug**，优化 RAG 切片/检索策略，提升知识提取质量。

---

## 二、问题全景 & 修复清单

### A. 图片内容提取（核心内容丢失）

| # | 场景 | 现状 | 修复方案 | 改动文件 |
|---|------|------|---------|---------|
| A1 | **小红书图文帖** | 只抓文字描述，图片干货丢失 ~70% | 从 SSR JSON 提取 `imageList` → `VisionOCR.ocr_multiple_images()` | `xiaohongshu.py` |
| A2 | **微信公众号文章** | `<img>` 标签被 `decompose()` 删除 | 先提取 `<img data-src>` 列表，再删标签，图片 OCR 内容嵌入正文 | `wechat.py` |
| A3 | **知乎回答/文章** | HTML→文本时 `<img>` 自然丢失 | 从 `content_html` 中提取 `<img src>` → OCR → 嵌入 | `general.py` |

**统一方案**：在 `RawContent` 增加 `images: List[str]` 字段，各抓取器提取图片 URL 列表，入库管线中对有图片的内容调用 OCR，将结果追加到正文末尾。

**成本控制**：每篇文章最多 OCR 前 5 张图片，每张 ~¥0.01，单篇最高 ¥0.05。

### B. Bug 修复

| # | 问题 | 修复方案 | 改动文件 |
|---|------|---------|---------|
| B1 | `process_media_ingest` 传 `file://` 给 `requests.get()` → **语音入库链路断了** | 改为直接传本地文件路径，`process_file_ingest` 区分本地路径和 URL | `callback.py` |
| B2 | 临时文件后缀硬编码 `.png`/`.mp3` | 改为 `suffix_map` 字典映射 | `callback.py` |
| B3 | 微信文章 <50 字判定 + 图片删除叠加 → 图片密集文章报错 | 先计算图片数，有图片时降低阈值到 10 字 | `wechat.py` |

### C. 企微消息扩展

| # | 消息类型 | 修复方案 | 改动文件 |
|---|---------|---------|---------|
| C1 | `file`（用户发 PDF/Word） | 通过 MediaId 下载 → Dispatcher 分发 | `callback.py` |
| C2 | `video`/`shortvideo` | 下载缩略图 → OCR + 返回提示"视频转录暂未支持" | `callback.py` |
| C3 | `event`（进入应用/关注） | 静默处理，不返回帮助提示 | `callback.py` |

## 三、RAG 策略优化

### 当前 RAG 管线

```
原始内容 → LLM清洗(截断8000字) → Markdown落库 → 切片(按标题/固定长度) → 双路索引
                                                        ↓                        ↓
                                                   ChromaDB向量            内存BM25
                                                        ↓                        ↓
                                              查询改写 → Wiki+Data并行召回 → RRF融合 → LLM答案
```

### 现存问题 & 优化方案

#### D1. 切片无 Overlap → 跨段落语义断裂

**现状**：`chunker.py` 按 `\n\n` 拆分，切片之间零重叠。一个概念恰好跨两段时，两个切片都不完整。

**优化**：增加 **150 字符滑动窗口重叠**。

```
现状:   [段落A           ] [段落B           ] [段落C           ]
优化后: [段落A       ][overlap][段落B       ][overlap][段落C      ]
                   ←150字→              ←150字→
```

**改动**：`chunker.py` 的 `_split_long_text()` 方法，拆分时保留前一个 chunk 末尾 150 字作为下一个 chunk 的开头。

#### D2. BM25 中文分词过于简单 → 专业术语召回差

**现状**：`bm25.py` 用单字+bigram 分词，"产品经理" 被拆成 `[产,品,经,理,产品,品经,经理]`，"品经" 是噪声。

**优化**：引入 `jieba` 分词，精确切分中文词汇。

```python
# 现状
tokens = cn_chars + cn_bigrams + en_words  # 单字+bigram

# 优化后
import jieba
cn_words = list(jieba.cut(cn_text))  # 精确分词
tokens = cn_words + en_words
```

**改动**：`bm25.py` 的 `tokenize()` 函数。`requirements.txt` 添加 `jieba`。

#### D3. LLM 输入截断 8000 字过于粗暴 → 长文尾部信息丢失

**现状**：`llm_cleaner.py:89` 行 `content[:8000]` 硬截断，一篇 1.5 万字长文的后半部分直接丢弃。

**优化**：对超长内容，提取**首尾各 3500 字 + 中间省略提示**，或分段清洗后合并。

```python
# 优化后
if len(content) > 8000:
    head = content[:3500]
    tail = content[-3500:]
    content = f"{head}\n\n[...中间内容省略约{len(content)-7000}字...]\n\n{tail}"
```

**改动**：`llm_cleaner.py` 的 `clean()` 方法。

#### D4. Summary Chunk 信息过少 → 泛化检索效果有限

**现状**：摘要切片只有 `标题 + summary + tags`，约 50-100 字。

**优化**：扩充 Summary Chunk，加入**正文前 300 字**作为上下文窗口，提升摘要切片的语义密度。

```python
# 优化后
summary_text = (
    f"标题: {doc_title}\n"
    f"摘要: {summary}\n"
    f"标签: {', '.join(tags)}\n"
    f"概览: {body[:300]}"
)
```

**改动**：`chunker.py` 的摘要切片构建逻辑。

#### D5. 知乎 API 异常被静默吞掉 → 问题难排查

**现状**：`general.py` 多处 `except Exception: pass`。

**优化**：改为 `except Exception as e: logger.warning(f"知乎API: {e}")`。

## 四、数据模型升级

### RawContent 扩展

```python
@dataclass
class RawContent:
    url: str
    title: str
    content: str
    author: str = ""
    source_platform: str = ""
    original_tags: Optional[List[str]] = None
    images: Optional[List[str]] = None      # 新增：图片URL列表
    content_type: str = "text"               # 新增：text/image_heavy/mixed
```

### 入库管线变化

```
现状:  抓取 → LLM清洗 → 落库 → 索引
                ↓
优化后: 抓取(+图片URL) → 图片OCR(前5张) → 拼接正文 → LLM清洗 → 落库 → 索引
```

新增步骤在 `main.py` 的 `ingest()` 和 `callback.py` 的 `process_ingest()` 中，抓取完成后、清洗之前插入图片 OCR 环节。

## 七、风险点

| 风险 | 应对 |
|------|------|
| 图片 OCR 增加 LLM 调用成本 | 限制每篇最多 5 张，预计月增 ¥2-3 |
| jieba 分词首次加载慢（~1s） | 延迟初始化，首次查询时加载 |
| 小红书图片 URL 可能有防盗链 | VisionOCR 已支持 URL 直接传入，Kimi Vision 会自行下载 |
| 切片 overlap 增加 ChromaDB 存储量 ~10% | 当前 1.17MB，影响可忽略 |
